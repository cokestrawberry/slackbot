# Deferred — 해결된 버그 조회 (트러블슈팅)

> **등록일:** 2026-04-22
> **우선순위:** Phase 2 후보
> **요청자:** 사용자

## 요구사항

특정 날짜 이후 해결된 버그를 조회하여 Slack 스레드에 Jira 링크와 함께 알려준다.

### 입력 예시 (자연어)

- `@지라봇 2026.03.11 부터 해결된 버그에 대해서 알려줘`
- `@지라봇 버그 2026.03.11`
- `@지라봇 2026.03.11 부터`

### 출력 예시

```
🐛 2026.03.11 이후 해결된 버그 (3건)

  • SLAC-7 로그인 페이지 /auth/login 500 에러 발생 (완료 2026.04.22)
  • SLAC-15 결제 금액 0원 표시 (완료 2026.04.20)
  • SLAC-22 세션 만료 시 무한 로딩 (완료 2026.03.15)

📊 총 3건 해결 / 8 SP 완료
```

## 구현 계획

### 라우팅

하이브리드 C안 적용:
- 키워드 매칭: `버그` + 날짜 패턴 감지 → 즉시 실행
- 자연어 fallback: Claude가 의도 파악 → "해결된 버그 조회" + 날짜 파라미터 추출

### 날짜 파싱

- 패턴: `yyyy.MM.dd`, `yyyy-MM-dd`, `yyyy/MM/dd`
- `~부터`, `~이후` 등의 조사 처리
- 날짜 없으면 기본값: 최근 7일 or 현재 스프린트 시작일

### 데이터 소스

- **DB 우선** (빠름): `SELECT * FROM issues WHERE issue_type = '버그' AND status_category = '완료' AND jira_updated >= :date`
- **Jira JQL fallback** (DB 동기화 안 된 경우): `type = Bug AND status = Done AND updated >= "2026-03-11"`

### 영향 범위

- `SlackEventController` — 라우팅에 버그 조회 분기 추가
- `IssueRepository` — 날짜 범위 + 타입 + 상태 쿼리 추가
- 새 서비스 또는 `ScrumReportService` 확장
- 날짜 파싱 유틸리티

## 선행 조건

- DB 자동 동기화가 정기적으로 돌아야 해결 시점이 정확
- `jira_updated` 필드가 "완료로 변경된 시점"과 정확히 일치하지 않을 수 있음
  → Jira changelog API로 정확한 상태 변경 시점 기록 필요 (추가 작업)
